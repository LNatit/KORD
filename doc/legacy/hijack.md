# 🕹️ 深入理解输入劫持 (Hijack Mode)

在传统的 Minecraft 冲突检测中，系统仅仅知道“两个动作绑定在了同一个键上”。但在复杂的模组生态中，按键冲突从来不是平等的：有些动作会默默共享输入，有些会霸道地吞噬事件，还有些会突然弹出 UI 抢走焦点。

为了精准诊断这些底层交互问题，Chord 系统引入了 **`HijackMode`（输入劫持模式）** 这一核心概念。它描述了一个动作在底层事件总线（Event Bus）上是如何处理物理输入流的。

---

## ⚙️ 第一部分：劫持模式的四种基础范式 (The 4 Modes)

在为按键编写语义配置时，模组作者必须根据按键对底层输入流的处理方式，将其归类为以下四种基础模式之一：

### 1. `PASSIVE` (被动接收 / P)
*   **定义**：良好的总线公民。动作被触发后，**不会阻断**底层事件向下传递（即不调用 `setCanceled(true)` 或返回 `CONSUME`）。
*   **特征**：允许多个动作“并发（Concurrent）”执行。
*   **典型案例**：原版的绝大多数操作（移动、跳跃）、非指向性的法术释放。

### 2. `CONSUME` (霸���吞噬 / C)
*   **定义**：物理输入的终结者。动作一旦触发，会**物理截断**底层事件流，导致排在后面的处理器永远收不到该按键的按下/抬起事件。
*   **特征**：具有绝对的排他性，常用于高优先级的核心交互或覆写原版逻辑。
*   **典型案例**：TACZ 的开火/瞄准、部分需要严格防误触的模组核心按键。

### 3. `MODIFIER` (修饰键监听 / M)
*   **定义**：组合键前缀。按下该键瞬间本身不产生任何实质性游戏动作，而是将系统推入一个“等待下一个按键输入”的监听状态。
*   **特征**：它在时间维度上拉长了输入状态机。单独按下不具备破坏性，但极度害怕被其他动作“污染”。
*   **典型案例**：FTB 连锁挖矿的按住修饰键、各种模组自定义的 `Alt`/`Ctrl` 替代键。

### 4. `TRIGGER` (模态触发 / T)
*   **定义**：焦点抢夺者。按下后会立即呼出一个 2D 面板或半透明轮盘，强制将游戏引擎的“输入焦点（Input Focus）”从 3D 世界转移到 UI 上。
*   **特征**：不仅消费物理按键，还会打断玩家当前的连贯操作流。
*   **典型案例**：法术选择轮盘（Radial Menu）、模组背包悬浮窗。

## 🏷️ 第二部分：碰撞真值表与冲突病症 (Conflict Tags)

当玩家将多个不同模式的动作绑定在同一个物理按键上时，Chord 系统的“劫持评估层（Hijack Layer）”会进行 2D 矩阵碰撞，并产出极其精准的**“病症标签（Conflict Tags）”**。

这套标签彻底摒弃了含糊的描述，直接点明底层发生了什么系统级故障：

| 状态组合 | 产生的冲突标签 (Tag) | 严重度定级 | 底层病症解读 |
| :--- | :--- | :--- | :--- |
| **P + P** | `(无)` 或 `SHARED_INPUT` | 🟢 **SAFE** | 事件成功多路复用。安全性交由后续的“动作矩阵层”判定。 |
| **C + 任何** | **`INPUT_INTERCEPTION`** (输入被拦截) | 🔴 **SEVERE** | `CONSUME` 像黑洞一样物理截断了事件，导致另一方陷入“饿死”状态，按键彻底瘫痪。 |
| **M + P/M** | **`COMBO_AMBIGUITY`** (组合歧义) | 🟡 **WARNING** | 当下按下可能安全，但随后按下其他键时，系统无法分辨玩家是想执行并发操作，还是想触发组合键，极易产生“幽灵触发（Ghost Input）”。 |
| **T + P/M** | **`ACTION_INTERRUPT`** (动作被打断) | 🔴 **SEVERE** | 玩家正试图保持连贯动作或按住修饰键，却被突然弹出的模态 UI 强行抢走了光标与焦点，操作流被粗暴打断。 |
| **T + T** | **`FOCUS_COLLISION`** (焦点碰撞) | 🔴 **SEVERE** | 两个 UI 引擎同时试图抢占屏幕的最高渲染层级和输入焦点，必定导致渲染重叠或死锁。 |

## 💡 进阶特性：不对称冲突与“合理状态覆盖”

劫持层的判定是**具有方向性（Asymmetrical）**的。

当发生 `CONSUME` 撞上 `PASSIVE` 时，系统默认会抛出 `INPUT_INTERCEPTION` (SEVERE) 错误。**但是，Chord 系统具备“状态包容性校验”智能：**

如果系统发现，加害者（`CONSUME`）带有非常严苛的前提状态（例如：`"states": {"tacz:weapon_held": "true"}`），而受害者（`PASSIVE`）是全局泛用的原版动作（例如：原版右键交互）。
系统会认为这是一种**符合预期的情境覆盖（Expected State Override）**。

此时，管线**不会报错**，而是优雅地返回：
*   对 `CONSUME` 侧：🟢 **SAFE**
*   对 `PASSIVE` 侧：🔵 **INFO** （提示玩家：在特定状态下，此动作会被接管）

通过这套逻辑，模组作者可以放心地使用 `CONSUME` 来接管原版操作，而不用担心引起 Chord 系统的误报！

# 🎼 Chord 语义系统贡献指南 (Semantic System Guide)

欢迎来到 Chord！传统 Minecraft 的按键冲突检测极其死板（只要同键位就标红），而 Chord 通过引入**“按键语义（Key Semantics）”**，让系统能够像人类一样理解按键组合是否合理。

作为社区贡献者，你的任务是为各大模组的按键编写语义配置文件（`keys.json`）。为了写出精准的配置，你需要理解以下 **5 个核心语义字段**，以及它们在 Chord 底层管线中是如何参与裁决的。

---

## 🗺️ 全局概览：Chord 是如何思考的？
当系统发现两个功能（A 和 B）被绑定在同一个物理按键上时，Chord 会按照以下顺序进行“漏斗式”筛选：
1. **短路拦截期**：如果它们在不同环境触发（`states`互斥），或为了同一目的（`intent`相同），直接放行，不报冲突！
2. **综合定级期**：如果必须发生碰撞，系统会综合评估它们的“输入劫持程度（`hijack_mode`）”、“操作形态（`modality`）”和“业务类型（`action_root`）”，最后给出最致命的那一个作为冲突警告。

以下是具体的字段说明与填写规范：

---

## 1. `states` (状态互斥组)
**代表什么：** 该按键生效的“先决条件 / 坐标系”。
**数据类型：** 键值对字典（Map<String, String>）

*   **管线层级：** 第 4 层 (State Mutex Layer)
*   **逻辑参与方式：** **一票否决冲突（SAFE）**。系统会检查 A 和 B 的状态字典，如果它们拥有相同的“状态组（Key）”，但“状态值（Value）”不同，说明它们**永远不可能在同一瞬间发生**，系统直接判定为完美兼容。
*   **填写指南与示例：**
    *   不要用它来描述动作，用它来描述**环境**。
    *   *例 1：原版步行 (W)* -> `"locomotion": "on_foot"`
    *   *例 2：Create 火车油门 (W)* -> `"locomotion": "driving"`
    *   *(由于 on_foot 和 driving 不同，这两个 W 键完美共存！)*

## 2. `hijack_mode` (输入劫持模式) ⚠️ *核心字段*
**代表什么：** 按下该键后，是否会“霸道”地阻断其他按键的触发，或强行改变玩家的注意力焦点？
**数据类型：** 枚举 (Enum)

*   **管线层级：** 第 6 层 (Input Hijack Layer)，并会前置影响第 5 层的意图判断。
*   **逻辑参与方式：** **恶性冲突提权**。它描述的是系统的“阻断感”。如果存在劫持，必定会引发 WARNING 或 SEVERE。
*   **可选值与填写指南：**
    *   `NONE` (默认)：正常的按键，按下去就执行，不吞输入也不弹复杂面板。
    *   `CONSUME_SELF` (自我消费)：**底层劫持**。按下后，它在底层吞掉了物理事件，导致排在它后面的原版按键完全失效。（*例：TACZ 的左键开火吞掉了原版左键破坏。同键位直接 SEVERE*）
    *   `CHORD_MODIFIER` (修饰键)：**幽灵劫持**。它本身没动作，但按下它会改变其他键的含义，变成组合键。（*例：F3 调试键。同键位直接 SEVERE，防止误触*）
    *   `MODAL_UI` (模态界面)：**焦点劫持**。它呼出了一个临时的轮盘、悬浮窗，需要玩家用鼠标或数字键去交互。（*例：Quark 悬浮背包、法术轮盘。与移动同键 WARNING，互相碰撞 SEVERE*）

## 3. `intent` (主观意图)
**代表什么：** 玩家按下这个键，脑子里想达到的“最终目的”是什么？
**数据类型：** 预定义字符串 / 枚举

*   **管线层级：** 第 5 层 (Intent Alias Layer)
*   **逻辑参与方式：** **同向合并放行**。如果 A 和 B 的 `intent` 相同，且双方都没有恶意的 `hijack_mode`（没吞输入），系统会认为这是“多个模组在用不同代码实现同一件事”，优先予以放行或降级。
*   **填写指南与示例：**
    *   能不填就不填，只有那些**高度通用、极易被多个模组重复实现**的基础动作才填。
    *   *常用 Intent：* `forward`, `backward`, `primary` (主行为/左键语义), `secondary` (副行为/右键语义), `open_menu`, `back` 等。

## 4. `modality` (操作模态)
**代表什么：** 玩家手指和这个按键的“时间互动关系”。
**数据类型：** 枚举 (Enum)

*   **管线层级：** 第 7 层 (Modality Layer)
*   **逻辑参与方式：** **防错位警告**。评估如果把这两种操作绑在一起，玩家手指会不会“打结”，或者导致状态机不同步。
*   **可选值与填写指南：**
    *   `PRESS` (按压)：按下瞬间即生效（例：切换第一/第三人称）。
    *   `HOLD` (长按)：按下开始，松开结束（例：原版疾跑、拉弓）。
    *   `TOGGLE` (开关)：按一下开启，再按一下关闭（例：某些模组的自动疾跑）。
    *   `CYCLE` (循环)：每按一次在几个状态间切一次（例：小地图模式切换）。
    *   *(管线逻辑：如果你把 HOLD 和 TOGGLE 绑在同一个键上，系统必报 SEVERE，因为按几次之后两者状态必然错乱。)*

## 5. `action_root` (动作根类别)
**代表什么：** 从业务逻辑上看，这究竟是一个什么性质的操作？
**数据类型：** 枚举 (Enum) - *注意：在 IN_GAME 和 IN_GUI 上下文中有两套不同的枚举库。*

*   **管线层级：** 第 8 层 (Action Root Matrix Layer)
*   **逻辑参与方式：** **查表定级**。经过前面所有漏斗后，系统会查询 N x N 矩阵表，得出这两种行为撞在一起的破坏力。
*   **填写指南与示例 (以 IN_GAME 为例)：**
    *   `MOVEMENT` (位移)：走、跳、潜行。与其他大部分动作同键都是 SAFE（边走边打很合理）。
    *   `ACT` (对世界干涉)：攻击、破坏、使用物品。
    *   `GUI_OPEN` (打开全屏界面)：打开背包、机器。与 `ACT` 撞在一起是 SEVERE（开箱子的瞬间不该开火）。
    *   `INFO` (轻量信息)：显隐血条、显示区块边界。
    *   `ABILITY` (能力)：施放法术、开启特殊能力。
    *   `SYSTEM` (系统)：截图、重载资源。与任何实操动作同键都是 SEVERE。

## 🛠️ 极简实战：我该如何为一个按键写语义？

当你拿到一个模组按键时，在脑海中快速问自己这 4 个问题：

1. **(状态)** 这个按键是不是只有在特定姿势（骑马/开车）下才有用？
    * *如果是，填 `states`。*
2. **(劫持)** 它会不会吞掉原来的按键？会不会弹出一个轮盘占掉数字键？
    * *如果是，选一个正确的 `hijack_mode`。*
3. **(意图)** 它是极其通用的“前后左右 / 主副行为”吗？
    * *如果是，填 `intent`，方便和其他模组共存。*
4. **(类别与时间)** 它属于哪类大动作（`action_root`），是一次性的还是长按的（`modality`）？
    * *填入对应的值，大功告成！*

> **最后提醒：Context is King (上下文为王)**
> 编写语义前，请务必分清这个按键是配置在 `ingame` (世界内游玩) 还是 `gui` (面板内操作) 节点下。Chord 为这两个世界准备了完全独立的判定矩阵！

# 按键冲突判定流水线 标准化总结格式（大白话版）
这份总结严格遵循代码执行的线性逻辑，用「固定模块+通俗比喻」的格式拆解全流程，**重点标注每个阶段的风险类型、传递规则，以及动态风险在后续环节的复用逻辑**，既适合非技术人员理解，也可直接用作流程文档。

---

## 前置说明&核心比喻
这套流水线的核心作用是：判断游戏中两个按键绑定，是否会在按下时发生功能冲突、冲突有多严重。
全程带着一本**「冲突信息收集本」**走完全流程：
- 每个阶段都会往本子上写：调试标签、静态/动态风险、是否终止流程的标记
- 前面阶段写的内容，后面所有阶段都能随时读取、修改、用于判定
- 一旦本子上打了「终止标记」，后续所有环节直接跳过，立刻出结果

### 术语大白话翻译
| 代码术语 | 通俗解释 |
|----------|----------|
| 主体/对手按键 | 本次拿来对比的两个按键绑定 |
| 物理键 | 键盘上实际按下的按键+Ctrl/Shift等修饰键组合（比如「Ctrl+F」和「F」是两个不同的物理键） |
| 生效场景（上下文） | 按键只有在这个场景下才会生效（比如「仅打开背包时生效」「仅在战斗时生效」） |
| 功能语义 | 按键按下去要实现的核心功能（比如「跳跃」「打开地图」「释放技能」） |
| 静态风险 | 固定不变的冲突标记，记在本子上就不会修改，仅用于最终结果判定 |
| 动态风险 | 可修改、可传递的「活风险」，后续阶段会读取它、调整它的严重等级，甚至用它决定是否终止流程 |
| 终止标记 | 一票否决标记，打了之后直接跳过后续所有环节 |

---

## 整体流程总览
全流程遵循**「从粗到细、逐层过滤」**的核心逻辑，先筛掉100%不会冲突的情况，再对可能冲突的场景做精细化校验，一共8个核心阶段，顺序绝对固定，不可颠倒。

---

## 分阶段详细流水线说明
每个阶段统一使用「阶段定位→核心动作→判定规则&流程走向→本阶段新增风险/标签→风险传递规则」的固定格式，确保逻辑清晰无遗漏。

### 阶段1：物理键一致性校验（入门第一关）
- **阶段定位**：全流程最基础的粗筛，先排除绝对不可能冲突的情况
- **核心动作**：检查两个按键是不是完全相同的物理键+修饰键组合，连按键都不一样，根本不会抢输入
- **判定规则&流程走向**：
    1. 若按键/修饰键有任何一项不一致：在收集本上记「硬件不匹配」调试标签，直接判定无冲突，**终止全流程**
    2. 若按键组合完全一致：通过本关，进入阶段2
- **本阶段新增**：仅静态调试标签，无任何风险
- **风险传递规则**：无，本阶段要么直接结束流程，要么不产生任何影响后续的内容

### 阶段2：用户自定义覆盖规则校验（用户优先级关）
- **阶段定位**：尊重用户自定义设置，用户规则优先级最高
- **核心动作**：检查用户有没有给这两个按键手动设置过「谁优先、是否冲突」的专属规则
- **判定规则&流程走向**：
    1. 若存在用户自定义规则：直接按用户设定的结果输出，**终止全流程**
    2. 若没有自定义规则：通过本关，进入阶段3
- **本阶段新增**：无额外标签/风险
- **风险传递规则**：无

### 阶段3：生效场景重叠校验（场景过滤关）
- **阶段定位**：第二道粗筛，排除「同按键但永远不会同时生效」的情况
- **核心动作**：遍历两个按键的所有「生效场景+功能」组合，一对一对检查它们的生效场景会不会撞在一起
- **判定规则&流程走向**：
    1. 某一对组合的生效场景完全不重叠：在收集本上记「场景路由」调试标签，跳过这对组合，检查下一对
    2. 某一对组合的生效场景有重叠（可能同时触发）：这对组合进入阶段4的精细化校验
    3. 所有组合检查完毕后，汇总所有结果
- **本阶段新增**：仅静态调试标签，无任何风险
- **风险传递规则**：无，仅筛选出需要进入后续精细化校验的组合

### 阶段4：状态互斥校验（触发状态过滤关，精细化校验第一关）
- **阶段定位**：第三道粗筛，排除「同场景同按键，但永远不会同时触发」的情况
- **核心动作**：对场景重叠的功能组合，检查两个功能的触发状态是否完全互斥（比如一个「蹲下时触发」，一个「站立时触发」，永远不会同时运行）
- **判定规则&流程走向**：
    1. 若触发状态完全互斥：在收集本上记「状态互斥」调试标签，这对组合判定无冲突，**终止该组合的后续校验**
    2. 若触发状态不互斥（有可能同时触发）：通过本关，进入阶段5
- **本阶段新增**：
    - 静态风险：静态调试标签
    - 动态风险：子集包含
- **风险传递规则**：无

### 阶段5：输入拦截规则校验（核心风险关，首个产生动态风险的阶段）
- **阶段定位**：第一个核心冲突判定环节，判断两个功能会不会抢输入，**产生的动态风险会直接影响后续阶段的判定**
- **核心动作**：检查两个功能有没有「截胡输入」的规则——也就是一个功能按下后，会抢走按键输入，不让另一个功能收到
- **判定规则&流程走向**：
    1. 先读取收集本中前面阶段的内容（本阶段为首个动态风险产生环节，无前置风险）
    2. 分情况判定：
        - 两个功能都开了「截胡输入」：在收集本上记**动态风险「竞态条件」**（两个功能抢输入，会出现随机失效）
        - 仅其中一个功能开了「截胡输入」：在收集本上记**动态风险「输入拦截」**（一个功能会抢走另一个的输入，导致其失效）
        - 两个功能有「子集包含」的特殊关系：在收集本上记静态风险「部分覆盖」，同时打**终止标记**，终止该组合的后续校验
    3. 若打了终止标记：该组合校验结束
    4. 未打终止标记：通过本关，进入阶段6
- **本阶段新增**：
    - 静态风险：部分覆盖（提示等级）
    - 动态风险：竞态条件、输入拦截（全程传递）
- **风险传递规则**：本阶段标记的**「输入拦截」动态风险**，会被阶段7主动读取、修改严重等级；所有动态风险都会跟随收集本进入后续所有环节

### 阶段6：重定向模式校验（输入转发规则关）
- **阶段定位**：固定规则匹配环节，判断输入转发逻辑是否会引发冲突
- **核心动作**：根据两个功能的「输入转发模式」，从预设的规则矩阵中匹配对应的冲突结果（比如一个功能会把输入转发给另一个，还是会直接吞掉输入）
- **判定规则&流程走向**：
    1. 按两个功能的重定向模式，匹配预设规则表中的冲突信息
    2. 将匹配到的**静态风险/调试标签**记录到收集本
    3. 若匹配结果要求终止流程：打**终止标记**，该组合校验结束
    4. 未打终止标记：进入阶段7
- **本阶段新增**：仅静态风险/调试标签，无动态风险
- **风险传递规则**：无，仅记录静态信息，不影响后续环节的判定逻辑

### 阶段7：高级功能语义校验（精细化冲突关，仅双高级功能开启，复用前置动态风险）
- **阶段定位**：最高阶的精细化校验，仅当两个功能都是「高级功能」时才会开启，**会主动读取前面阶段的动态风险，调整判定逻辑和风险等级**
- **前置开启条件**：两个功能必须均为带高级属性的功能，否则直接跳过本阶段，进入最终结果汇总
- 本阶段分为2个固定子步骤，顺序不可颠倒：

#### 子步骤7-1：意图匹配校验
- **核心动作**：检查两个功能的使用意图是否一致（比如两个功能都是「打开背包」，就算有拦截，也不算恶性冲突）
- **判定规则&流程走向**：
    1. 先读取收集本中，阶段5标记的**「输入拦截」动态风险**
    2. 若两个功能的意图有共享/完全一致：
        - 若存在前置的「输入拦截」动态风险：直接将该风险的严重等级从「警告」降级为「提示」，同时打**终止标记**，终止该组合的后续校验
        - 若没有前置动态风险：在收集本上记**动态风险「意图共享」**，标记是否完全一致
    3. 打了终止标记：该组合校验结束
    4. 未打终止标记：进入子步骤7-2
- **本环节新增**：动态风险「意图共享」，可修改前置的「输入拦截」动态风险
- **风险传递规则**：修改后的动态风险会保留到最终结果中

#### 子步骤7-2：操作模态校验
- **核心动作**：检查两个功能的操作模式是否冲突（比如一个是「按住持续触发」，一个是「点击一次触发」）
- **判定规则&流程走向**：
    1. 按两个功能的操作模态，从预设的规则矩阵中匹配对应的冲突信息
    2. 将匹配到的**静态风险/调试标签**记录到收集本
    3. 本环节结束，进入最终结果汇总
- **本环节新增**：仅静态风险/调试标签
- **风险传递规则**：无

### 阶段8：预留功能-资源读写冲突校验（暂未启用）
- **阶段定位**：预留的高阶校验环节，目前未加入流水线
- **核心动作**：检查两个功能是否会同时读写同一个游戏资源（比如同时修改玩家属性），区分「并发写」「读写冲突」的严重等级

---

## 最终结果输出规则
1. 所有「生效场景+功能」组合的校验结果，全部汇总到「冲突信息收集本」中
2. 收集本里的所有调试标签、静态风险、动态风险（含修改后的严重等级），全部整合到最终的冲突结果中
3. 最终结果按风险的最高严重等级，判定按键组合的冲突等级：无冲突→轻微提示→警告→严重冲突

---

## 核心规则关键提醒
1. **单本全流程通用**：全程只有一本「冲突信息收集本」，所有环节的内容都可跨阶段读取、修改
2. **动态风险是活的**：前面环节标记的动态风险，会在后续环节被复用、调整，甚至直接决定流程是否终止
3. **终止标记一票否决**：任何环节打了终止标记，后续所有环节直接跳过，不会再做额外校验
4. **逐层过滤逻辑**：先粗筛掉绝对无冲突的情况，再对高风险场景做精细化校验，最大程度减少无效判断
